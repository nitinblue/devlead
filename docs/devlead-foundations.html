DevLead Foundations

Two guarantees that make AI-assisted development governable. Everything else builds on these.

The Two Foundations

Foundation 1: CLAUDE.md is always derived from devlead_docs/

What it means: The LLM's instruction file (CLAUDE.md) is never hand-written. It's auto-generated by reading every file in devlead_docs/ and rendering them into one document.

Why it matters: If the LLM's instructions can drift from the project's actual state, governance breaks. This foundation guarantees they're always in sync.

How it works:

  • Read order — derived from which files exist
  • Discipline rule — derived from _living_decisions.md
  • Commands — derived from commands/*.md descriptions
  • File categories — derived from actual files present
  • Routing table — derived from _routing_table.md
  • Current state — derived from hierarchy + intake status

BUILT bootstrap.py generate_section() renders from files. devlead init writes it.

Foundation 2: No code without project management flow

What it means: The LLM physically cannot edit or create files unless it has an active intake entry (work item) set as focus. Hard block — exit code 2.

Why it matters: Without this, the LLM writes code freely, says "done," and nobody can trace what it did or why. This foundation makes every code change traceable.

How it works:

  • PreToolUse hook fires on every Edit/Write/MultiEdit
  • DevLead checks _intake_*.md for entries with status: in_progress
  • If none found → exit 2 → tool call BLOCKED
  • If found → exit 0 → tool call proceeds
  • Every check logged to _audit_log.jsonl

BUILT gate.py enforces. Default mode: hard. Configurable via devlead.toml.

What Needs to Be Built Next

Priority 1: Stop Hook (session end automation)

Without this, session continuity depends on Claude choosing to run commands. The Stop hook should auto-run devlead resume + devlead dashboard + kpi.record_session(). This makes session handoff guaranteed, not voluntary.

Priority 2: UserPromptSubmit Hook (intent routing)

The routing table is passive — it's in CLAUDE.md but Claude might drift. A UserPromptSubmit hook would fire on every user message, match against routing table triggers, and inject the matched route as a systemMessage. Same mechanism as skills but always active.

Priority 3: Intake → Hierarchy Promotion

The pipeline from scratchpad → intake works. The pipeline from intake → hierarchy is manual. A devlead promote --to hierarchy command would close this gap and make the full flow automated.

Priority 4: Verify Runner (auto-check TTO boxes)

The dashboard runs verify commands and shows pass/fail. But it doesn't update the hierarchy checkboxes. A devlead verify command should run all verify commands, check passing TTO boxes, and update convergence. That's DevLead marking work done — not Claude.

Priority 5: Document DAG Propagation

SOT blocks have receives_from and migrates_to fields defining relationships between files. Nothing reads them. When _routing_table.md changes, CLAUDE.md should auto-regenerate. When hierarchy changes, resume should auto-update. The DAG engine reads SOT relationships and triggers downstream updates.

TTO mapping: TTO-015 (DAG engine), TTO-016 (routing→CLAUDE.md), TTO-017 (hierarchy→resume), TTO-018 (K_DRIFT KPI)

Priority 6: Session Experience Analysis

DevLead should analyze the conversation itself — not via survey, but by reading session patterns. Frustration signals: how many times did the user say "stop", "wrong", "why". Fudging signals: how many times Claude said "done" vs how many TTOs verified. Session sentiment derived from conversation, not opinion. Feeds K_EXPERIENCE KPI.

TTO mapping: TTO-042 (self-correction), TTO-043 (K_INCONSISTENCY KPI)

The Product Vision

What DevLead is: A governance tool that makes any LLM accountable. The LLM follows predefined paths (routing table), traces every code change to a work item (intake → hierarchy), and proves what's done via automated verification (Definition of Done). The user — who may not be a coder — opens one HTML file and sees the truth.

What DevLead is NOT: A wrapper around Claude. DevLead is MD-driven — the routing table, hierarchy, and governance rules are all markdown. Any LLM that reads markdown can be governed. Python adds computation (KPIs, reports, convergence math) but governance works without it.

Two tiers:

  • Free: MD-only governance. Copy devlead_docs/ into your project. Works with any LLM.
  • Paid: Python-enhanced. pip install devlead. Adds KPIs, dashboard, hooks, audit log, convergence math.
