Context

Development was frozen after Session 3 because Nitin wants to pivot from breadth (new features) to depth (cohesiveness, flexibility, configurability, agility, gracefulness). This document analyses two phase objectives and proposes concrete mechanisms for each.

  1. Memory model cohesion. Clutter-less. Leak-free. Full lineage. The memory layer should be a tight, traceable graph of content with clear ownership and lifecycle.
  2. Enforcement design. How DevLead gracefully overrides LLM behavior. Specifically: how to keep _intake_*.md files as the source of truth such that nothing gets developed without a traceable entry.

Non-goals. No new features. No new breadth. Every proposal below tightens what already exists.

3.3 Where the graph breaks

Gap 1: _intake_*.md entries cannot be promoted to _project_hierarchy.md. No command exists. Everything stops at intake today.
Gap 2: No mechanism ties a specific code edit to a specific TTO. Once the store layer ships, this becomes the main enforcement surface.
Gap 3: Decisions land in _resume.md §3 by hand. No automation promotes them to _living_decisions.md. Over time, §3 grows unbounded (see L7).
Gap 4: Scratchpad entries never retire even after promotion. The cross-reference is one-way.

4. Tightening proposals

4.1 Proposal A — Source-of-truth declaration blocks

Every file in devlead_docs/ opens with a standardized metadata block that DevLead reads to understand the file's role, owner, and lifecycle. Hand-editable; parseable by a tiny Python module.

<!-- devlead:sot
  purpose: "Session handoff cursor + locked decisions + open threads"
  owner: "claude+user"
  canonical_for: ["session_memory", "locked_decisions"]
  lineage:
    receives_from: ["_scratchpad.md (via triage)"]
    migrates_to: ["_living_decisions.md (on Sprint close)"]
  lifetime: "permanent"
  bloat_cap_lines: 200
  last_audit: "2026-04-14"
-->

DevLead reads the block to answer questions like:

  • Which file is canonical for locked decisions? (whichever has locked_decisions in canonical_for)
  • Where does content in _resume.md migrate to when it matures? (migrates_to)
  • Is _resume.md over its bloat cap? (compare line count vs bloat_cap_lines)
  • When was this file last sanity-checked? (last_audit)

Parser: simple regex. Block is a markdown HTML comment wrapping YAML-ish key/value lines. Stdlib only. Roughly 80 lines in a new src/devlead/sot.py module.

Install impact: all scaffold templates gain an SOT block. Existing files need a one-time backfill (done during rollout item 2).

4.2 Proposal B — Lineage metadata on every entry

Intake entries already have source, promoted_to, promoted_at, origin. Extend the same pattern across all unit types that can be referenced.

Entry typeCurrent metadataAdd

Scratchpad entry (_scratchpad.md)entry_id (date+slug), cross-refs in bodyFormalize cross-refs as a promoted_to: [intake-ids] structured field (dedupe-safe)

Benefit: a new /devlead verify-links command can walk the entire graph and catch broken references, orphans, and stale content automatically.

4.3 Proposal C — Migration protocol with zero-loss guarantee

Section 18.4 of _resume.md already designs this mechanism; it's not built. Build it.

  1. User or Claude proposes: "Section X of _resume.md should move to _design_section_Y.md."
  2. DevLead computes a content hash of the source section.
  3. DevLead checks whether the destination file exists and contains matching content (by hash).
  4. If yes: safe to remove from source. Append to _migration_log.jsonl. Remove.
  5. If no: migration is blocked. Error: "destination does not hold this content; write it first, then retry."

This is the one place in DevLead where strict mode is always on: information loss is unrecoverable, so it can never be a warning.

Commands:

  • /devlead migrate <source-section> --to <dest-file>
  • /devlead migrate rollback <migration-id>
  • /devlead migrate list (shows all past migrations)

Impact on file bloat: combined with the bloat_cap_lines SOT field, migration becomes the pressure-release valve. _resume.md stays lean because mature content flows out.

4.4 Proposal D — _resume.md discipline via bloat nudges

Every time _resume.md is modified, DevLead checks its line count against the SOT bloat_cap_lines value (default 200). If over, DevLead emits a warning listing sections that are candidates for migration (largest first, oldest first).

Advisory in v1. In v1.1 it becomes a Stop-hook check that fires at session end, reminding Claude to propose migrations before closing.

Example warning:

devlead: _resume.md is at 620 lines (cap 200).
Sections over 50 lines:
  §4  "The hierarchy (with math)"           120 lines  [stale per L1]
  §18 "Session Resume first-class feature"  180 lines  [stale per L2]
  §19 "Project Maturity Score"               90 lines  [migrate candidate]
Run /devlead migrate §4 --to _design_section1.md to relocate.

5. Enforcement spectrum — Level 0 through 5

There are six degrees of how rigorously DevLead can override LLM behavior, ordered from softest to hardest. Each has different graceful-ness and different implementation cost.

LevelMechanismHow it overrides ClaudeGraceful?CostCurrent?
0Convention only CLAUDE.md rule text. Claude self-polices by reading the rule at session start. VeryZeroYes (today)
1Soft nudging Claude-injected reminders mid-session; e.g. Claude prints "Remember: no code without intake trace" before an edit. VeryLow (CLAUDE.md text + Claude discipline)Partial
2Hook + systemMessage Claude Code PreToolUse hook fires before Edit/Write. DevLead script inspects context, returns a systemMessage in the hook's JSON output. Claude reads it and self-corrects. Exit 0 — no hard block. GracefulMedium (hook + Python check + session state file)No
3Soft block via exit 1 Same hook but exits 1 if the check fails. Claude cannot proceed without addressing it; user can explicitly override. MediumMediumNo
4Hard block via exit 2 Hook exits 2. Claude cannot proceed. Only user intervention unblocks. Not gracefulMediumNo (and v2 forbids exit 2 by default)
5Pre-commit hook Git pre-commit hook. Independent of Claude Code. Rejects commits that don't reference an intake/TTO ID. User can --no-verify but visibly. MediumLowNo
Recommendation for v1: Level 2. Graceful. Uses native Claude Code hooks. Warns without blocking. Supports a config knob to upgrade to Level 3 for users who want stricter behavior.
(UPDATE 2026-04-15) The six-level spectrum was collapsed in the locked decision to three modes: warning (Level 2 in this table), soft (Level 3), hard (Level 4). Default is warning. Levels 0/1/5 were dropped as either too weak or too late. Note that as of Session 3 close, nothing in this spectrum has been implemented yet — the discipline rule lives in CLAUDE.md text only (effectively Level 0). The gate engine (src/devlead/gate.py) is deferred to a future session.

Why not Level 0/1? Purely convention-based. Relies on Claude's diligence. No programmatic signal when the rule is bypassed. No audit trail.

Why not Level 3/4? Hard blocks contradict the _resume.md §3 locked decision: "Nothing in v2 ever exits 2 unless strict mode is explicitly enabled." Level 2 is the warn-by-default implementation of that rule. Level 3 becomes an opt-in.

Why not Level 5? Only catches violations at commit time, way too late. Dark code has already been written.

6. Recommended enforcement for v1 — Level 2 detailed

The mechanism has four moving parts: the hook wiring, the CWI tracker, the Python check, and the configuration surface. Each is small; they compose into graceful enforcement.

6.1 The hook wiring

Claude Code supports four hooks today: SessionStart, UserPromptSubmit, PreToolUse, Stop. For intake discipline, we care about PreToolUse with matcher Edit|Write|MultiEdit.

// .claude/settings.json (DevLead-owned plumbing; users don't edit)
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write|MultiEdit",
        "hooks": [
          {
            "type": "command",
            "command": "devlead gate PreToolUse",
            "timeout": 5
          }
        ]
      }
    ]
  }
}

/devlead init writes this file (v1.1 feature; currently {"hooks": {}} by design until the gate engine ships).

6.2 The CWI tracker

(UPDATE 2026-04-15: SUPERSEDED.) This subsection proposed a separate _session_state.json for CWI tracking. The locked decision went the other way: focus is tracked via the status: in_progress field inside the intake entry body itself — no parallel session-state file, no session.py module, no CWI concept. See §0 Resolution Log row Q3 and the _living_decisions.md row "2026-04-14 – Focus via intake entry status (not a separate file)". The subsection below is preserved as the original reasoning archive; do not implement it.

CWI = Current Working Intake. The intake entry (or set of entries) that Claude is actively implementing. Without a CWI, the gate cannot verify traceability.

Tracked in a new file: devlead_docs/_session_state.json.

{
  "session_id": "2026-04-14T15:30:00Z",
  "cwi": ["FEATURES-0003", "BUGS-0001"],
  "cwi_set_at": "2026-04-14T15:35:00Z",
  "cwi_set_by": "user"
}

Commands to manage CWI:

  • /devlead focus <intake-id> [<intake-id> ...] — set CWI to one or more entries. Can be called multiple times in a session.
  • /devlead focus clear — clear CWI (for non-code sessions: docs work, capture-only).
  • /devlead focus show — print current CWI.

CWI is session-scoped by default but persists across sessions via the JSON file. Claude is expected to set it before editing code (documented in CLAUDE.md).

6.3 The Python check

A new module: src/devlead/gate.py. Entry point: check_pretooluse(hook_input: dict) -> dict.

# pseudocode
def check_pretooluse(hook_input):
    tool = hook_input["tool_name"]
    file_path = hook_input["tool_input"].get("file_path", "")

    # Skip meta-edits (DevLead's own files, intake files themselves)
    if _is_devlead_meta(file_path):
        return {"continue": True}

    # Skip exempt paths (docs, config, tests, markdown) per config
    if _is_exempt(file_path):
        return {"continue": True}

    session_state = _read_session_state()
    cwi = session_state.get("cwi", [])

    if not cwi:
        _audit("gate_warn", tool=tool, file=file_path, reason="no_cwi")
        return {
            "continue": True,  # warn mode, never blocks
            "systemMessage": (
                "DevLead discipline: no current working intake is set. "
                "Before editing code, run `/devlead focus <intake-id>` "
                "or create an intake entry via "
                "`/devlead ingest --from-scratchpad ... --forced`."
            )
        }

    # Optional check: CWI entries mention the file being edited
    if _config_enabled("enforcement.trace_file_edits"):
        matching = [id for id in cwi if _entry_mentions_file(id, file_path)]
        if not matching:
            _audit("gate_warn", tool=tool, file=file_path,
                   cwi=cwi, reason="file_not_in_cwi")
            return {
                "continue": True,
                "systemMessage": (
                    f"DevLead discipline: {file_path} is not mentioned in "
                    f"any current CWI entry ({cwi}). Verify trace or refocus."
                )
            }

    _audit("gate_pass", tool=tool, file=file_path, cwi=cwi)
    return {"continue": True}

Key properties:

  • Never exits non-zero in warn mode. Claude always gets the tool call; it just reads the systemMessage.
  • Exempts DevLead's own files and non-code paths (configurable).
  • Writes every check to _audit_log.jsonl (pass, warn, or block).

6.4 What Claude sees

When Claude is about to edit src/devlead/cli.py without a CWI, Claude Code shows:

DevLead discipline: no current working intake is set.
Before editing code, run `/devlead focus <intake-id>` or
create an intake entry via
`/devlead ingest --from-scratchpad ... --forced`.

Claude Code still runs the Edit tool (warn mode). But Claude reads the message, understands the nudge, and can choose to respond: "Before I make this edit, let me first set focus or capture the intake..." and then call the appropriate command.

This is the graceful path: enforcement happens as conversational pressure, not hard failure. The cost of noncompliance is not a broken tool; it is a visible audit-log event and a nudge that Claude (and the user reading Claude's output) cannot miss.
